<p>A couple months ago I wrote about Moltbot—how it's essentially Claude Code with a Telegram wrapper and a security posture that would make a pentester cry. The short version: no sandboxing, plaintext credentials, exposed admin panels, and supply chain attacks via its skills library. I ended that post saying I'd keep my AI agents on a short leash.</p>

<p>That wasn't idle commentary. I immediately started building something that actually did the job properly.</p>

<p>This is that build.</p>

<h2 class="section-heading">The Problem Statement</h2>

<p>What I actually wanted was simple to describe, hard to build: an AI assistant that I can talk to through Discord, that runs inference locally on my GPU, that can't touch files or network resources I haven't explicitly allowed, and that doesn't phone home to a cloud API with my conversations.</p>

<p>Moltbot's failure wasn't the concept—it was the execution. An always-on AI assistant that can take action is genuinely useful. But "always-on" and "can take action" are exactly the attributes that demand a serious security model, and Moltbot treats them as an afterthought. The researchers at Bitdefender and Cisco flagged it; the exposed instances on Shodan confirmed it.</p>

<p>The irony is that NVIDIA, of all companies, already had a proper solution. I just had to actually deploy it.</p>

<h2 class="section-heading">What NemoClaw Is</h2>

<p>NemoClaw is NVIDIA's OpenClaw agent running inside their OpenShell sandbox framework. OpenClaw is an agentic AI assistant—it can write code, run shell commands, browse the web, manage files. OpenShell is the containment layer: a k3s-based runtime that enforces network policies and Landlock filesystem isolation at the kernel level.</p>

<p>Landlock is a Linux Security Module that restricts filesystem access independent of process permissions. Even if the agent is running as a privileged user, Landlock enforces what directories it can actually touch. This is meaningfully different from "I put it in a Docker container and called it sandboxed." It's kernel-enforced isolation.</p>

<p>The network side is handled through declarative policy files. You define exactly which external hosts the sandbox can reach, and the OpenShell gateway enforces it. By default, the sandbox can reach the NVIDIA cloud inference API and nothing else. You add rules explicitly.</p>

<p>For inference, NemoClaw integrates with NVIDIA's NIM microservices. NIM packages AI models in optimized containers with TensorRT-LLM backends. Instead of paying per token to a cloud API and sending my queries to someone else's servers, the model runs on my RTX 5000 Ada sitting under my desk.</p>

<h2 class="section-heading">The Stack</h2>

<p>Here's what the full system looks like:</p>

<pre><code>Discord #tpk-agent
      │
      ▼
discord-bridge/bot.py  (Python, discord.py)
      │  SSH subprocess per message
      ▼
openclaw agent  (inside tpk-agent sandbox)
      │
      ▼
openclaw gateway  (ws://127.0.0.1:18789)
      │
      ▼
nim-proxy.py  (request rewriter, port 8009)
      │
      ▼
NIM container  (Nemotron Nano 4B v1.1, TRT-LLM, RTX 5000 Ada)
</code></pre>

<p>Discord messages hit a Python bot that SSHes into the sandbox and runs <code>openclaw agent --message "..." --json</code>. The response comes back as JSON and gets posted to the channel. The sandbox itself handles inference by routing through the openclaw gateway, which reaches the NIM container via a compatibility proxy I wrote.</p>

<p>That proxy is where a lot of the actual work happened.</p>

<h2 class="section-heading">The Real Work: NIM Compatibility</h2>

<p>OpenClaw is designed to work with OpenAI-compatible APIs, but it sends several parameters that NIM's TRT-LLM backend flat-out rejects with HTTP 400s. Some are format differences, some are parameters NIM doesn't support, and one caused a problem I didn't see coming.</p>

<p>The format issues were straightforward but numerous: <code>tool_choice</code> needs to be a plain string, not an object; <code>message.content</code> needs to be a plain string, not a typed array; <code>max_completion_tokens</code> must become <code>max_tokens</code>; and <code>store: false</code> needs to be stripped entirely. The proxy rewrites all of this on the fly.</p>

<p>The tool schema problem was more of a pain. OpenClaw sends full function schemas with every request. NIM's TRT-LLM backend compiles these into a Finite State Machine index for structured output—a thoughtful design, but compilation takes 10 to 15 minutes per unique schema, completely blocks the GPU, and restarts every time NIM restarts.</p>

<p>The fix was to strip tool definitions from requests at the proxy layer. With <code>STRIP_TOOLS=1</code>, the proxy removes <code>tools</code> and <code>tool_choice</code> before forwarding, NIM sees a plain completion request, and it responds in under two seconds.</p>

<p>The tradeoff is that the model can't use tools. For my use case—general assistant through Discord—that's acceptable. The 4B model isn't going to be reliably running complex multi-step tool chains anyway. I can re-enable tool forwarding by setting <code>STRIP_TOOLS=0</code> and accepting the slow cold start.</p>

<h2 class="section-heading">The Model Problem</h2>

<p>I want to be honest about this part, because the original NemoClaw documentation is written around the 120B Nemotron Super model running on NVIDIA cloud infrastructure. I'm running the 4B Nemotron Nano model locally.</p>

<p>The 4B model is capable for general Q&A and conversation. It is not capable of reliably following OpenClaw's full system prompt, which is approximately 27,000 characters of agent instructions, workspace context, and tool documentation. With the full prompt, the model would occasionally output <code>NO_REPLY</code>—OpenClaw's special silent response token—instead of actually responding. It was getting confused by conditional instructions in the tool descriptions that referenced the token in edge cases.</p>

<p>I tried 8B first. It OOM'd. TensorRT-LLM at bf16 precision needs more VRAM headroom than my 16GB card has once WSL2 and the OS take their share. The 4B fits comfortably. Hardware limitation, not a stack one.</p>

<p>The fix for the NO_REPLY problem was to deny all tools at the OpenClaw config level (<code>tools.deny: ["*"]</code>), which removes their descriptions from the system prompt entirely. The resulting system prompt is around 16,000 characters, which the model handles correctly. Responses are fast—typically under two seconds from Discord message to reply.</p>

<h2 class="section-heading">The Security Model (vs. Moltbot)</h2>

<p>This is the part that actually matters to me. Let me compare the two directly.</p>

<p><strong>Credential storage:</strong> Moltbot stores API keys in plaintext files under <code>~/.clawdbot/</code>. NemoClaw's sensitive config (gateway token, inference API key) lives inside the OpenShell sandbox container. The host machine has an <code>.env</code> file for startup variables, which I've added to <code>.gitignore</code>. The difference is that commodity infostealers already know where Moltbot keeps its keys; the OpenShell container is not a standard target.</p>

<p><strong>Sandboxing:</strong> Moltbot runs with your full user permissions by default. OpenShell enforces Landlock filesystem restrictions at the kernel level—the agent can only access directories explicitly granted in its policy. It can't read my SSH keys, my browser cookies, or anything else on the host that I haven't explicitly allowed.</p>

<p><strong>Network access:</strong> Moltbot's network access is whatever your OS allows. OpenShell uses a YAML network policy file that I commit to the project repository. Nothing else. If the model decides to try to exfiltrate data, the kernel-enforced policy blocks it at the network layer.</p>

<p><strong>Inference:</strong> Moltbot calls out to Claude, GPT, or whatever API key you've configured. Every query leaves your machine. With NIM, inference runs locally—Nemotron Nano on my GPU, never touching the internet. My conversations, my system state, my questions don't leave the host.</p>

<p><strong>Supply chain:</strong> OpenClaw has a curated skills library with significantly less community contribution than Moltbot's ClawdHub. That's sometimes frustrating (fewer integrations) but it's the right tradeoff in the early days of agentic integration. I'm not installing skills from anonymous contributors until the open source community is stronger in this area.</p>

<p>Is it perfect? No. The 4B model is running with a simplified toolset. The sandbox configuration took real effort to get right. And if you hand the agent enough permissions, it can still do damage within those permissions. Security is about reducing attack surface, not eliminating it.</p>

<p>But it's meaningfully better than Moltbot's "we don't guarantee this is secure, use at your own risk" posture.</p>

<h2 class="section-heading">The Setup Reality</h2>

<p>I want to be clear that this wasn't a clean one-command install. The documented NemoClaw setup is designed for the NVIDIA cloud model where the 120B Nemotron handles OpenClaw's complex prompts without issue. Running it with a local 4B model required finding and fixing a series of compatibility problems.</p>

<p>The NIM proxy took a few iterations. The FSM compilation problem isn't documented anywhere—I found it by watching NIM's GPU utilization sit at 100% for fifteen minutes and then checking what TensorRT-LLM was actually doing. The NO_REPLY problem required reading OpenClaw's minified JavaScript to find the token, then tracing back why the 4B model was triggering it on edge-case conditional instructions in the tool descriptions. This was a heavy Claude Code session to untangle.</p>

<h2 class="section-heading">Hypothetical Use Cases</h2>

<p><em>The scenarios below are speculative. Any use of agentic AI in a real investigative or legal context should be reviewed and approved by qualified legal and forensic professionals before deployment. AI output is not a substitute for trained human analysis.</em></p>

<p>I work in digital forensics and incident response, and one workflow that comes to mind is warrant returns from electronic service providers. Investigators regularly receive data packages from companies like Google, Apple, Meta, and Discord in response to legal process. These returns are typically layered archives—sometimes encrypted, sometimes nested—that unpack into sprawling directory trees of JSON, SQLite databases, CDR spreadsheets, and account metadata. The first challenge is often just figuring out what's in there.</p>

<p>A local agent with shell access could hypothetically automate the unpack chain given the provided credentials, inventory the file types, and surface what data categories are present before an investigator starts digging. From there, natural language queries against call detail records ("who did this number contact most frequently in March"), schema discovery on unfamiliar SQLite databases, or cross-referencing device identifiers across returns from multiple providers become conversational rather than manual.</p>

<p>The critical word is <em>local</em>. ESP returns contain victim and suspect data that cannot go to a cloud API—full stop. The data sovereignty argument that applies to homelab curiosity becomes a hard requirement in an investigative context. A model running on your hardware, with no network egress, and a sandbox scoped to only the files relevant to the case is a meaningfully different proposition than pasting evidence into a chat interface backed by someone else's servers.</p>

<p>For classroom use, the architecture also makes a more honest demo than a polished cloud product. Showing students the full pipeline—model, sandbox, network policy—makes data sovereignty and isolation concrete rather than abstract. You can demonstrate what the model <em>can't</em> reach, which is often the more important lesson.</p>

<p>On the hobbyist side, the value is simpler: it's persistent and it's mine. Context about ongoing projects, homelab state, tools in progress—it carries between sessions without me wondering what a third party logs or retains.</p>

<h2 class="section-heading">The Result</h2>

<p>After all that, what I have is an AI assistant—I named it Ahab—that runs entirely on hardware I own, processes queries in under two seconds, responds in Discord, and can't touch anything on my system I haven't explicitly granted it access to.</p>

<p>The Discord interface works exactly how I wanted. Message to <code>#tpk-agent</code>, typing indicator while the SSH subprocess runs, response appears. Context maintained within sessions. It knows who it is because that's in workspace files I control, not a system prompt I'm trusting a cloud API to follow.</p>

<p>The security model is one I can actually defend: kernel-enforced filesystem isolation, declarative network policy as code, local inference with no data leaving the host.</p>

<p>Is a 4B model going to replace Claude Code for complex software engineering tasks? No. But that's not what this is for. It's an always-on assistant that can answer questions, help me think through problems, and run simple tasks—without requiring me to hand my credentials to an unsandboxed process or trust that a cloud API isn't logging my queries.</p>

<p>The next step is upgrading the model. Once I have a larger GPU budget, I'll point this at a 70B model running through Ollama and see how the tooling holds up. The architecture is ready for it—the proxy, the gateway config, the Discord bridge—all of it is model-agnostic. Swap the model ID and the NIM container image, and everything else stays the same.</p>

<p>Until then, Nemo runs on four billion parameters and answers in two seconds, and I know exactly what it can and can't access.</p>

<p>That's the short leash I was talking about.</p>
